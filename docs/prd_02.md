# PRD 02 — 레시피 생성 (재료 목록 → 추천 레시피)

- 작성일: 2026-07-23
- 상태: Draft
- 관련 단계: 2/3
- 의존성: PRD 01의 재료 목록 출력을 입력으로 사용

## 개요
1단계(PRD 01)에서 인식되고 사용자가 확인/수정한 재료 목록을 입력으로, `NVIDIA: Nemotron 3 Nano Omni (free)` 모델을 텍스트 전용으로 호출하여 만들 수 있는 레시피를 추천한다.

## 목표
- 보유 재료만으로(또는 최소한의 추가 재료로) 만들 수 있는 레시피를 여러 개 추천
- 조리법을 실제로 따라할 수 있는 수준으로 구체화

## 범위
**In scope**
- 재료 목록 기반 레시피 생성 API
- 레시피 목록 UI (제목/재료/조리법/예상 시간)

**Out of scope**
- 사용자별 저장/프로필 (→ PRD 03)
- 결제, 장보기 리스트 연동 등 부가 기능

## 사용자 흐름
1. PRD 01의 최종 재료 목록(사용자가 보정한 버전)을 입력으로 받음
2. 백엔드가 재료 목록 + (선택) 사용자 선호도/제약을 프롬프트로 구성
   - PRD 03 완료 전까지는 선호도를 매 요청마다 폼으로 입력받거나 생략
3. OpenRouter 텍스트 생성 호출 (동일 모델, 이미지 입력 없음)
4. 응답을 레시피 카드 형태로 파싱해 표시 (예: 3개 추천)
5. 사용자가 마음에 드는 레시피를 선택 (저장은 PRD 03에서 처리)

## 기능 요구사항
| ID | 내용 |
|---|---|
| FR-1 | `POST /api/recipes/generate` — body: `{ items: [...], preferences?: {...} }` |
| FR-2 | 프롬프트: "가진 재료만으로 또는 최소한의 추가 재료로 만들 수 있는 레시피 N개를 JSON으로 생성" |
| FR-3 | 레시피 구조: `title`, `have`(보유 재료), `need`(부족한 재료), `steps`, `est_time_min`, `difficulty` |
| FR-4 | 부족한 재료는 UI에서 명확히 구분 표시 |
| FR-5 | 재시도/에러 처리는 PRD 01과 동일한 지수 백오프 패턴 재사용 |

## 데이터 모델
```json
{
  "recipes": [
    {
      "title": "계란 볶음밥",
      "have": ["계란", "밥"],
      "need": ["파"],
      "steps": ["팬에 기름을 두르고 파를 볶는다", "밥과 계란을 넣고 볶는다"],
      "est_time_min": 15,
      "difficulty": "easy"
    }
  ]
}
```

## 기술 노트
- 텍스트 전용 호출이라 이미지 인식보다 무료 워커 부하가 낮을 것으로 예상되나, PRD 01과 동일한 재시도 로직을 그대로 적용
- 1단계(이미지 인식)와 2단계(레시피 생성) API 호출은 분리 유지 — 관심사 분리 및 단계별 독립 테스트/캐싱을 위해

## 성공 기준 (Acceptance Criteria)
- 재료 목록 입력 시 최소 1개 이상의 유효한 레시피 반환
- 조리법이 순서대로 실제로 따라할 수 있는 수준으로 구체적
